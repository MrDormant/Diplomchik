ГИД ПО ПРОЕКТУ
===============

Название проекта: веб-сервис для компании промышленного строительства.
Стек: FastAPI, PostgreSQL, SQLAlchemy, Alembic, HTML, CSS, JavaScript, Docker Compose.


1. ОБЩАЯ ИДЕЯ ПРОЕКТА
---------------------

Проект представляет собой веб-сервис для строительной/инжиниринговой компании.

Пользователь может:

- открыть сайт компании;
- посмотреть информацию о компании;
- сделать предварительный расчет объекта;
- зарегистрироваться или войти;
- оформить заявку;
- посмотреть свои заявки в личном кабинете;
- выбрать решение из библиотеки видов работ;
- посмотреть готовые проекты.

Менеджер или администратор может:

- войти в панель менеджера;
- видеть все заявки клиентов;
- менять статус заявки;
- оставлять комментарий для клиента;
- смотреть пользователей;
- добавлять решения в каталог;
- добавлять готовые проекты.

Главная идея проекта: показать полный путь заявки от пользователя до менеджера:

1. Пользователь делает расчет.
2. Пользователь отправляет заявку.
3. Заявка сохраняется в базе данных.
4. Менеджер получает уведомление.
5. Менеджер обрабатывает заявку.
6. Клиент видит статус заявки в личном кабинете.


2. ОБЩАЯ АРХИТЕКТУРА
--------------------

Проект разделен на две основные части:

- backend;
- frontend.

При этом приложение остается монолитным: frontend и backend лежат в разных папках,
но запускаются одним FastAPI-сервисом.

Это значит:

- backend отвечает за API, базу данных, бизнес-логику и безопасность;
- frontend отвечает за HTML-страницы, стили, клиентский JavaScript и изображения;
- FastAPI отдает HTML-страницы и статические файлы;
- JavaScript на страницах обращается к API по адресам вида /api/v1/...

Такой подход удобен для дипломного проекта, потому что архитектура понятная,
но не перегружена микросервисами или отдельным frontend-сервером.

4. BACKEND: ГДЕ ЧТО ЛЕЖИТ
-------------------------

Основная папка backend:

backend/app/

Это Python-пакет FastAPI-приложения.


4.1. backend/app/main.py

Это главная точка входа приложения.

В этом файле:

- создается объект FastAPI;
- подключается общий API-роутер;
- подключается раздача статических файлов;
- описываются маршруты для HTML-страниц.

Примеры страниц, которые отдает FastAPI:

- /;
- /library.html;
- /projects.html;
- /calculator.html;
- /cabinet.html;
- /admin.html.

Frontend лежит отдельно, но FastAPI отдает его пользователю.

HTML берется из:

frontend/html/

Статические файлы доступны по адресу:

/assets/...

Например:

/assets/css/style.css
/assets/js/index.js
/assets/library-covers/ALUDAR.jpg


4.2. backend/app/api/v1/

Здесь находится API проекта.

Общий префикс API:

/api/v1

То есть все backend-запросы начинаются с /api/v1.

Основные файлы endpoints:

backend/app/api/v1/endpoints/calculate.py
  Endpoint для предварительного расчета объекта.

backend/app/api/v1/endpoints/auth_users.py
  Регистрация, вход, получение текущего пользователя.

backend/app/api/v1/endpoints/projects.py
  Работа с проектами пользователя.

backend/app/api/v1/endpoints/requests.py
  Работа с заявками клиента.

backend/app/api/v1/endpoints/public.py
  Публичные endpoints, доступные без авторизации.

backend/app/api/v1/endpoints/solutions.py
  Каталог решений и категорий.

backend/app/api/v1/endpoints/showcase.py
  Готовые проекты.

backend/app/api/v1/endpoints/admin.py
  Административные и менеджерские функции.

backend/app/api/v1/router.py
  Собирает все endpoints в один общий роутер.


5. СЛОИ BACKEND
---------------

Проект разделен на слои. Это важная часть архитектуры.


5.1. models

Папка:

backend/app/models/

Здесь находятся SQLAlchemy-модели.

Модель - это Python-класс, который описывает таблицу базы данных.

Основные модели:

user.py
  Пользователи системы.

project.py
  Проекты и расчеты пользователя.

service_request.py
  Заявки клиентов.

solution.py
  Решения из библиотеки видов работ и категории решений.

showcase.py
  Готовые проекты.

company.py
  Информация о компании.

user_log.py
  Лог действий пользователей.

Что можно сказать дипруку:

"Модели описывают структуру таблиц базы данных и связи между сущностями.
Через SQLAlchemy приложение работает с таблицами как с Python-объектами."


5.2. schemas

Папка:

backend/app/schemas/

Здесь находятся Pydantic-схемы.

Схемы нужны для:

- проверки входящих данных;
- описания структуры ответа API;
- автоматической документации Swagger.

Например, если пользователь отправляет заявку, схема проверяет,
что пришли нужные поля и они правильного типа.

Что можно сказать:

"Схемы отделяют формат данных API от моделей базы данных. Это делает API
предсказуемым и защищает от некорректных входных данных."


5.3. crud

Папка:

backend/app/crud/

CRUD - это слой простых операций с базой данных:

- create;
- read;
- update;
- delete.

Файлы:

project.py
  Работа с проектами.

request.py
  Работа с заявками.

showcase.py
  Работа с готовыми проектами.

solution.py
  Работа с решениями и категориями.

user.py
  Работа с пользователями.

Что можно сказать:

"CRUD-слой вынесен отдельно, чтобы API-роуты не содержали прямую логику
работы с базой данных."


5.4. services

Папка:

backend/app/services/

Здесь находится бизнес-логика.

Файлы:

calculator.py
  Формула предварительного расчета.

request_service.py
  Сценарий создания заявки.

email.py
  Отправка email-уведомлений.

admin_service.py
  Логика менеджера: изменение статуса, заметки и т.д.

Что можно сказать:

"Сервисы описывают бизнес-сценарии. Например, создание заявки - это не просто
запись в таблицу, а последовательность действий: проверить данные, создать
заявку, связать ее с пользователем или гостем и отправить уведомление."

Папка:

Что можно сказать:

"В core вынесены общие механизмы приложения: конфигурация, безопасность и

Frontend работает через обычные HTTP-запросы.

В JavaScript используется fetch().

Примеры запросов:

POST /api/v1/calculate
  Выполнить расчет.

POST /api/v1/users
  Зарегистрировать пользователя.

POST /api/v1/login
  Войти и получить токен.

GET /api/v1/me
  Получить текущего пользователя.

GET /api/v1/solutions
  Получить решения для библиотеки.

GET /api/v1/showcase-projects
  Получить готовые проекты.

GET /api/v1/requests
  Получить заявки текущего клиента.

GET /api/v1/admin/requests
  Получить все заявки для менеджера.

PATCH /api/v1/admin/requests/{id}
  Изменить статус и комментарий заявки.


8. ОСНОВНОЙ СЦЕНАРИЙ ПОЛЬЗОВАТЕЛЯ
---------------------------------

1. Пользователь открывает главную страницу:

http://localhost:8000/

2. FastAPI отдает index.html.

3. Браузер загружает:

/assets/css/style.css
/assets/js/index.js
/assets/js/site-nav.js

4. Пользователь вводит параметры объекта:

- длина;
- ширина;
- высота;
- тип объекта;
- тип каркаса.

5. JavaScript отправляет запрос:

POST /api/v1/calculate

6. Backend рассчитывает:

- площадь;
- ориентировочную стоимость;
- срок;
- массу материала.

7. Результат возвращается в JSON.

8. JavaScript отображает результат на странице.

9. Пользователь нажимает "Оформить заявку".

10. Если пользователь не вошел:

- открывается форма гостевой заявки;
- пользователь вводит имя и телефон;
- заявка создается без аккаунта.

11. Если пользователь вошел:

- создается проект;
- создается заявка, связанная с пользователем и проектом;
- пользователь переходит в личный кабинет.

12. Менеджер видит заявку в панели.

13. Менеджер меняет статус.

14. Клиент видит обновленный статус в кабинете.


9. АВТОРИЗАЦИЯ И РОЛИ
---------------------

В проекте есть роли:

client
  Обычный пользователь.

manager
  Менеджер, который обрабатывает заявки.

admin
  Администратор, расширенная роль.

Авторизация работает через JWT.

Сценарий:

1. Пользователь вводит email и пароль.
2. Frontend отправляет их на /api/v1/login.
3. Backend проверяет пароль.
4. Если пароль верный, backend возвращает access_token.
5. Frontend сохраняет токен в localStorage.
6. Для защищенных запросов frontend отправляет заголовок:

Authorization: Bearer <token>

Пароли не хранятся открытым текстом. Они хешируются.

Что можно сказать:

"Авторизация построена на JWT. После входа клиент получает токен и использует
его для доступа к защищенным endpoints."

14. DOCKERFILE
--------------

Dockerfile описывает сборку API.

Основные шаги:

1. Используется python:3.12-slim.
2. Создается рабочая директория /app.
3. Устанавливаются зависимости из requirements.txt.
4. Копируются backend и frontend.
5. Устанавливается PYTHONPATH=/app/backend.
6. Открывается порт 8000.
7. Запускаются миграции и uvicorn.

Зачем PYTHONPATH:

Пакет FastAPI называется app и лежит внутри backend/app.
Чтобы Python в контейнере мог импортировать app.main, нужно добавить backend
в путь поиска модулей.

Команда запуска внутри контейнера:

alembic -c backend/alembic.ini upgrade head && uvicorn app.main:app --host 0.0.0.0 --port 8000

17. КОРОТКИЙ ТЕКСТ ДЛЯ РАССКАЗА
-------------------------------

Можно сказать так:

"Мой проект - это веб-сервис для компании промышленного строительства.
Пользователь может открыть сайт, сделать предварительный расчет объекта,
зарегистрироваться и оставить заявку. Менеджер видит заявки в отдельной панели,

Архитектура разделена на backend и frontend. В backend находится FastAPI,

Я выбрал такую архитектуру, потому что она достаточно простая для дипломного
проекта, но при этом правильно разделяет ответственность между слоями
приложения."


18. ОТВЕТЫ НА ВОЗМОЖНЫЕ ВОПРОСЫ
-------------------------------

Вопрос: Почему проект разделен на backend и frontend?

Ответ:
"Так проще поддерживать проект. Backend отвечает за данные, API и бизнес-логику,
а frontend - за интерфейс. При этом они запускаются вместе, что удобно для
учебного проекта."


Вопрос: Почему не React?

Ответ:
"Для задач проекта достаточно обычного HTML, CSS и JavaScript. Интерфейс не
настолько сложный, чтобы подключать React. Я разделил HTML и JS по файлам,
поэтому frontend все равно остается аккуратным."


Вопрос: Почему FastAPI?

Ответ:
"FastAPI удобен для создания REST API, имеет автоматическую документацию Swagger,
поддерживает Pydantic-валидацию и хорошо подходит для учебных и практических
backend-проектов."


Вопрос: Зачем Docker?

Ответ:
"Docker позволяет запускать проект одинаково на разных компьютерах. Не нужно
вручную настраивать PostgreSQL, Mailpit и окружение Python. Достаточно выполнить
docker compose up --build."


Вопрос: Зачем Alembic?

Ответ:
"Alembic нужен для управления схемой базы данных. Если структура таблиц меняется,
это фиксируется миграцией. Так базу можно воспроизводимо создать или обновить."


Вопрос: Зачем Mailpit?

Ответ:
"Mailpit используется как тестовый почтовый сервер. Когда создается заявка,
приложение отправляет email, но письмо не уходит наружу, а появляется в локальном
интерфейсе Mailpit."


Вопрос: Где хранится бизнес-логика?

Ответ:
"В папке backend/app/services. Например, расчет находится в calculator.py,
создание заявки - в request_service.py, отправка писем - в email.py."


Вопрос: Где описана база данных?

Ответ:
"Таблицы описаны в backend/app/models через SQLAlchemy-модели. Миграции Alembic
лежат в backend/alembic/versions."


Вопрос: Как защищены пароли?

Ответ:
"Пароли не хранятся в открытом виде. Они хешируются. При входе пароль проверяется
по хешу, а пользователю выдается JWT-токен."


Вопрос: Как менеджер получает доступ к заявкам?

Ответ:
"Доступ проверяется по роли пользователя. Для endpoints менеджера используется
проверка роли manager или admin."


19. САМОЕ ВАЖНОЕ, ЧТО НУЖНО ПОМНИТЬ
-----------------------------------

Если нужно объяснить проект очень коротко:

1. Это сайт и CRM-часть для строительной компании.
2. Клиент делает расчет и отправляет заявку.
3. Заявка сохраняется в PostgreSQL.
4. Менеджер обрабатывает заявку в панели.
5. Клиент видит статус в личном кабинете.
6. Backend написан на FastAPI.
7. Frontend - обычные HTML/CSS/JS.
8. База - PostgreSQL.
9. Миграции - Alembic.
10. Запуск - Docker Compose.


20. КОМАНДЫ, КОТОРЫЕ ЛУЧШЕ ЗНАТЬ
--------------------------------

Запуск:

docker compose up --build

Остановка:

docker compose down

Остановка с удалением базы:

docker compose down -v

Загрузка тестовых данных:

docker compose exec api python backend/scripts/seed.py

Открыть сайт:

http://localhost:8000/

Открыть Swagger:

http://localhost:8000/docs

Открыть Mailpit:

http://localhost:8025/

Локальный запуск без Docker:

uvicorn app.main:app --reload --app-dir backend

Применить миграции вручную:

alembic -c backend/alembic.ini upgrade head


21. ФИНАЛЬНАЯ ФОРМУЛИРОВКА
--------------------------

"Проект построен как FastAPI-монолит с четким разделением на backend и frontend.
Backend отвечает за API, авторизацию, базу данных, заявки, бизнес-логику и
отправку уведомлений. Frontend отвечает за страницы сайта и взаимодействие с
пользователем. PostgreSQL хранит данные, Alembic управляет схемой базы,
Docker Compose поднимает все окружение для демонстрации."
